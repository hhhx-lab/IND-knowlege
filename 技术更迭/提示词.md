## 阶段 1：TBox 动态本体构建（架构师提示词）

**目标**：输入代表性的 Markdown 样本，输出 IND 领域的 Schema（实体、属性、公理）。

```markdown
### Role
你是一位深耕生物医药领域 20 年的“药政合规官”与“本体论建模专家”。你擅长从繁杂的 CTD 申报资料中抽象出严密的逻辑结构。

### Task
请分析以下 Markdown 格式的 IND 申报资料片段，提取该领域的 TBox（抽象模型）。输出需包含：
1. **实体类 (Classes)**：识别核心对象（如：药物、适应症、不良反应、临床方案、风险控制措施等）。
2. **对象属性 (Object Properties)**：定义类之间的谓语关系（如：[药物]-发起了-[临床试验], [不良反应]-被收录于-[参考文献]）。
3. **领域公理 (Axioms)**：定义业务逻辑规则（如：若 A 是“严重不良事件”，则 A 必须具有“医学评价结果”属性）。

### Constraints
- 请识别 IND 模块 1.8（药物警戒）与 1.3.4（方案）之间的隐性逻辑。
- 采用 JSON 格式输出 Schema 结构，以便程序读取。
- 公理描述需具体，能指导后续推理。

### Input Data

```

## 阶段 3：知识融合与规范化（清洗专家提示词）

**目标**：消除歧义，合并同义词，将实体映射到国际标准。

```markdown
### Role
你是一位“医学信息学专家”，负责知识图谱的实体消歧（Entity Resolution）与规范化（Normalization）。

### Context
目前图谱中存在大量从不同文档中提取的散落实体。你需要将它们合并并对齐。

### Task
1. **实体对齐**：判断“TQB2858”、“TQB2858 注射液”、“该研究药物”是否指向同一个 `Drug` 实例。若是，请输出合并指令。
2. **术语映射**：将所有的“不良反应 (AE)”实体，尝试映射到 MedDRA（监管活动医学词典）标准术语。例如，将“拉肚子”映射为“腹泻 (Diarrhoea)”。
3. **关系消减**：去除逻辑上重复或冲突的关系。

### Output Format
输出为一份“合并与映射清单”，列出：
- 原实体名称 -> 标准实体名称 (Standard_ID)
- 映射理由
```

## 技术方案的实施建议

* **Prompt 迭代**：建议您先在您的 **1.8 章节（药物警戒）** 上测试阶段 2 的提示词。1.8 章节的结构化程度高，包含大量的缩略语、流程和时间限制，是检验抽取精度的最佳试金石。
* **多模型协同**：
    * **阶段 1 和 4**：建议使用性能最强的模型（如 Gemini 1.5 Pro），因为它需要处理复杂的逻辑和长上下文。
    * **阶段 2 和 3**：可以使用您本地部署的小模型集群（如 Llama 3 或类似模型），它们在处理这种“按表填空”的任务时效率极高。
